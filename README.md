## venv(virtual environment)
- グローバルな環境でpip installをすると、グローバル用の場所にパッケージがインストールされてしまう。
- プロジェクトごとに異なるバージョンの同一パッケージを利用したい場合などにvenvの真価が発揮される
- python3 -m venv myenv とすると、以下のようなディレクトリが作られる
```
myenv/
├── bin/                  # Windowsでは Scripts/
│   ├── python            # システムのpythonへのsymlink（またはコピー）
│   ├── python3           # 同上
│   ├── pip
│   └── activate          # シェルスクリプト
├── lib/
│   └── python3.x/
│       └── site-packages/  # ← ここに隔離してインストールされる
└── pyvenv.cfg            # 設定ファイル（これが最重要）
```
- myenv/bin/activateスクリプトを実行すると、一時的に環境変数PATHを書き換える
- PATHの先頭に`/path/to/myenv/bin`を追加することで、優先的にmyenv以下のpythonのsymlinkが使われるようになる

### パッケージの隔離ができるのはなぜか？
Pythonインタプリタは起動時に「自分がどこにインストールされているか（sys.prefix）」を、実行された python バイナリの位置から逆算します。具体的には、バイナリと同じ階層やその親に pyvenv.cfg があるかをチェックします。

- myenv/bin/python を実行すると、すぐ上の myenv/ に pyvenv.cfg を発見する
- すると sys.prefix を myenv に設定し、site-packages の探索先を myenv/lib/python3.x/site-packages/ に切り替える
- 結果として pip install も import もこのディレクトリを見るようになる

## pipとpoetry
- pip はインストラーであって、依存管理ツールではない。言われたものを入れるだけ。(pip install での指定や、requirements.txt通りのものをインストールするだけ)
- poetry は、宣言・解決・再現・隔離の統合ツール
  - 宣言: pyproject.tomlに欲しい物を宣言する
  - 依存解決: pandasが要求するnumpy、そのnumpyが要求する...をすべてたどって、すべてが矛盾しないようインストールされる
  - 再現: 依存関係の計算結果をpoetry.lockにすべて書き出す。その後、poetry installを別環境で実行すると、lockを見て、寸分違わず環境を再現できる
  - 隔離: 勝手に裏で、venvの作成・管理をしてくれていて、グローバルな環境を汚したり、他の環境と干渉したりすることがない


## 依存解決の方針の違い
- pipはそもそも解決しようとしないので、あとからインストールされたバージョンで上書きされる（あと勝ち）例えば、pandasが要求するnumpyが1.2.3だったとして、その後に、numpy1.2.0を指定されてインストールされるとそれで上書きされる。
- poetryの場合は、インストールする前に、要求されたパッケージのすべての依存関係を洗い出し、矛盾が生じれば、インストール自体をしない
- node_modulesなどでは、同一パッケージを複数インストールできる。あるパッケージが依存するパッケージは依存元のパッケージ以下に入れ子でインストールする仕組みになっている

## pyproject.tomlとpoetry.lockの違い
- pyproject.tomlに指定するのは要求するパッケージのバージョン範囲
- lockファイルの方では、その時点で解決した際に確定した各パッケージの固定バージョンを記録している。よって完全に同じ環境を実行タイミングに関係なく再現できる
  - バージョンだけでなく、かくぱ

~2.1.0マイナーを上げない>=2.1.0,<2.2.0
>=2.1,<3.0明示的な範囲そのまま
